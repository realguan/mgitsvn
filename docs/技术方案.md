# mgitsvn - VS Code 插件技术方案

## 1. 项目概述

### 1.1 背景

公司项目采用微服务架构，代码由 SVN 托管。开发人员习惯使用 Git 工作流，因此通过 `git-svn` 进行代码管理。当前存在以下痛点：

| 痛点 | 描述 |
|------|------|
| 分散管理 | 每个服务独立仓库，需逐一操作 |
| 重复劳动 | 同步、拉取、提交都需进入各目录执行 |
| 分支管理复杂 | 跨多仓库的分支切换和 worktree 管理繁琐 |

### 1.2 目标

开发一个 VS Code 插件 `mgitsvn`，实现：

- **一键批量操作**：同步、拉取、提交所有服务代码
- **分支管理**：跨多仓库统一切换分支
- **Worktree 支持**：批量创建/删除 worktree，实现多分支并行开发

---

## 2. 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 开发语言 | TypeScript | VS Code 插件标准语言，类型安全 |
| 插件框架 | VS Code Extension API | 官方原生支持 |
| Git 操作 | simple-git | 成熟的 Node.js Git 库，支持 git-svn |
| 配置管理 | JSON + VS Code Settings | 用户友好，支持工作区级别配置 |
| UI 组件 | VS Code TreeView + Webview | 原生体验 + 复杂交互支持 |

## 4. 核心模块设计

### 4.1 配置管理模块 (ConfigurationManager)

**配置文件**: `.mgitsvn.json` (工作区根目录)

```json
{
  "rootDir": "/home/user/projects/my-project",
  "projects": [
    {
      "name": "service-config",
      "path": "service-config",
      "svnUrl": "svn://svn.example.com/repos/project/trunk/service-config"
    },
    {
      "name": "service-web",
      "path": "service-web",
      "svnUrl": "svn://svn.example.com/repos/project/trunk/service-web"
    }
  ],
  "defaultBranch": "dev",
  "worktreeBaseDir": "/home/user/projects"
}
```

**核心接口**:

```typescript
interface MgitsvnConfig {
  rootDir: string;
  projects: ProjectConfig[];
  defaultBranch: string;
  worktreeBaseDir: string;
}

interface ProjectConfig {
  name: string;
  path: string;
  svnUrl: string;
  enabled?: boolean;
}
```

### 4.2 仓库管理模块 (RepositoryManager)

**职责**:
- 管理多个 git-svn 仓库的生命周期
- 提供批量操作能力

**核心方法**:

```typescript
class RepositoryManager {
  // 初始化/发现项目仓库
  async discoverRepositories(): Promise<Repository[]>;

  // 批量 git svn rebase
  async rebaseAll(options?: RebaseOptions): Promise<BatchResult>;

  // 批量 git svn dcommit
  async dcommitAll(options?: DcommitOptions): Promise<BatchResult>;

  // 获取所有仓库状态
  async getStatusAll(): Promise<RepositoryStatus[]>;

  // 批量切换分支
  async checkoutAll(branch: string): Promise<BatchResult>;
}
```

### 4.3 Worktree 管理模块 (WorktreeManager)

**职责**:
- 批量创建/删除 worktree
- 管理 worktree 与分支的映射关系

**核心方法**:

```typescript
class WorktreeManager {
  // 为所有项目创建 worktree
  async createWorktreeAll(branch: string): Promise<BatchResult>;

  // 删除指定分支的所有 worktree
  async removeWorktreeAll(branch: string): Promise<BatchResult>;

  // 列出所有 worktree
  async listWorktrees(): Promise<WorktreeInfo[]>;

  // 在 VS Code 中打开 worktree 工作区
  async openWorktreeWorkspace(branch: string): Promise<void>;
}
```

### 4.4 Git-SVN 适配器 (GitSvnAdapter)

**职责**:
- 封装底层 git-svn 命令
- 处理命令执行和错误

```typescript
class GitSvnAdapter {
  // git svn clone
  async clone(svnUrl: string, targetPath: string, options?: CloneOptions): Promise<void>;

  // git svn rebase
  async rebase(repoPath: string): Promise<RebaseResult>;

  // git svn dcommit
  async dcommit(repoPath: string): Promise<DcommitResult>;

  // git svn fetch
  async fetch(repoPath: string): Promise<void>;
}
```

|------|------|-----------|

| `mgitsvn.statusAll` | 查看所有仓库状态 | `Ctrl+Shift+S` |

| `mgitsvn.createWorktree` | 批量创建 worktree | - |
| `mgitsvn.removeWorktree` | 批量删除 worktree | - |
| `mgitsvn.openWorktree` | 打开 worktree 工作区 | - |
| `mgitsvn.cloneProject` | 克隆新的服务项目 | - |

```
[mgitsvn: 5/5 synced | dev] [↓2 ↑3]
```

- 显示同步状态
- 当前分支
- 待拉取/待推送数量

### 5.4 通知与进度

- 批量操作时显示进度条
- 操作完成后显示汇总通知
- 错误发生时提供详细日志链接

## 6. 数据流设计

### 6.1 批量 Rebase 流程

```
用户点击 "Rebase All"
        │
        ▼
┌───────────────────┐
│ 读取配置获取项目列表 │
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 并发执行 rebase    │◄─── 使用 Promise.allSettled
│ (可配置并发数)      │      保证部分失败不影响整体
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 收集结果并汇总      │
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 更新 UI 状态       │
│ 显示成功/失败详情   │
└───────────────────┘
```

### 6.2 Worktree 创建流程

```
用户输入分支名
        │
        ▼
┌───────────────────────────────┐
│ 验证分支名合法性               │
└─────────────┬─────────────────┘
              │
              ▼
┌───────────────────────────────┐
│ 创建 worktree 根目录           │
│ ${worktreeBaseDir}/${branch}  │
└─────────────┬─────────────────┘
              │
              ▼
┌───────────────────────────────┐
│ 遍历所有项目                   │
│ git worktree add -b ${branch} │
└─────────────┬─────────────────┘
              │
              ▼
┌───────────────────────────────┐
│ 生成 .code-workspace 文件      │
│ 方便一键打开整个 worktree      │
└─────────────┬─────────────────┘
              │
              ▼
┌───────────────────────────────┐
│ 提示用户是否立即打开新工作区    │
└───────────────────────────────┘
```

## 7. 错误处理策略

| 错误类型 | 处理策略 |
|---------|---------|
| 单个仓库 rebase 冲突 | 标记该仓库状态为冲突，继续处理其他仓库，最后汇总提示 |
| git-svn 命令不存在 | 启动时检测，提示用户安装 git-svn |
| 网络超时 | 提供重试机制，支持断点续传 |
| 权限不足 | 明确提示具体仓库和所需权限 |
| 配置文件错误 | 提供配置向导，自动发现仓库 |

## 10. 技术风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| git-svn 命令执行慢 | 用户体验差 | 并发执行 + 进度显示 + 后台任务 |
| 跨平台兼容性 | Windows 用户无法使用 | 测试 Windows/Mac/Linux，处理路径差异 |
| 大量仓库时性能问题 | 响应缓慢 | 分批处理 + 虚拟列表 + 缓存 |
| git-svn 版本差异 | 命令不兼容 | 检测版本，适配主流版本 |

## 12. 总结

本方案设计了一个模块化、可扩展的 VS Code 插件架构，核心解决多仓库 git-svn 的批量操作痛点。采用 TypeScript 开发，使用 VS Code 原生 API 保证良好的用户体验。通过清晰的分层设计，便于后续功能扩展和维护。
